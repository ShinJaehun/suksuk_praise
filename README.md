# 쑥쑥칭찬통장

쑥쑥칭찬통장은 교실에서 학생에게 칭찬을 기록하고, 칭찬을 바탕으로 쿠폰을 발급하며, 학생이 자신의 칭찬과 쿠폰을 직접 확인할 수 있도록 만든 Rails 기반 교실 보상 관리 웹 애플리케이션입니다.

종이 칭찬통장이나 스티커판처럼 교사가 일일이 기록하고 확인해야 하는 방식에서 벗어나, 교사와 학생이 각자의 화면에서 칭찬, 쿠폰, 메시지 상태를 즉시 확인할 수 있도록 설계했습니다. 특히 교실 수업 중 태블릿이나 공용 기기를 활용해 학생이 PIN으로 간단히 로그인하고 자신의 쿠폰을 확인하거나 사용 요청을 보낼 수 있는 흐름을 목표로 합니다.

---

## 이 저장소에 대하여

이 저장소는 초등학교 교실에서 사용할 수 있는 **칭찬 기록 + 쿠폰 보상 + 학생 포털** 시스템을 구현한 Rails 프로젝트입니다.

쑥쑥칭찬통장의 핵심 방향은 다음과 같습니다.

- 교사는 교실을 만들고 학생을 등록합니다.
- 교사는 학생에게 칭찬을 줄 수 있습니다.
- 학생은 자신이 받은 칭찬과 쿠폰을 확인할 수 있습니다.
- 칭찬을 많이 받은 학생은 칭찬왕 후보가 됩니다.
- 교사는 칭찬왕이나 특정 학생에게 쿠폰을 발급할 수 있습니다.
- 학생은 발급받은 쿠폰의 사용을 요청하고, 교사는 이를 승인할 수 있습니다.
- 교사와 학생은 교실 맥락 안에서 메시지를 주고받을 수 있습니다.
- 교실 화면에서는 학생별 알림 아이콘으로 쿠폰 사용 요청과 읽지 않은 학생 메시지를 확인할 수 있습니다.

---

## 왜 만들었나

교실에서 칭찬과 보상을 운영하다 보면 다음과 같은 어려움이 생깁니다.

- 칭찬 기록이 종이, 스티커판, 구두 안내 등으로 흩어집니다.
- 학생이 자신이 받은 칭찬이나 쿠폰을 즉시 확인하기 어렵습니다.
- 쿠폰을 발급하고 사용 처리하는 과정이 교사에게 반복 업무가 됩니다.
- 여러 학생의 쿠폰 사용 요청이나 메시지를 한눈에 파악하기 어렵습니다.
- 담임교사, 관리자, 학생의 권한과 화면을 구분하기 어렵습니다.
- 교실 수업 중 학생이 빠르게 로그인해서 참여하기 어렵습니다.

쑥쑥칭찬통장은 이런 문제를 해결하기 위해, 교실 단위로 칭찬과 쿠폰을 관리하고 학생이 자기 화면에 직접 접근할 수 있는 구조로 만들었습니다.

---

## 사용자 역할

쑥쑥칭찬통장은 세 가지 사용자 역할을 기준으로 동작합니다.

| 역할 | 설명 |
| --- | --- |
| 관리자 | 전체 교실과 교사, 라이브러리 쿠폰 템플릿을 관리합니다. |
| 교사 | 자신이 담당하는 교실, 학생, 칭찬, 쿠폰, 메시지를 관리합니다. |
| 학생 | 교실별 PIN 로그인으로 접속해 자신의 칭찬, 쿠폰, 메시지를 확인합니다. |

권한 관리는 Devise 인증과 Pundit 정책을 기반으로 합니다. 관리자는 전체 범위에 접근할 수 있고, 교사는 자신이 담당하는 교실만 관리하며, 학생은 자신과 자신이 속한 교실 맥락에 제한됩니다.

## 주요 기능

### 1. 교실 관리

교사는 자신의 교실을 만들고 관리할 수 있습니다. 관리자는 교실을 만들 때 담당 교사를 배정하거나, 기존 교실의 담당 교사를 조정할 수 있습니다.

교실에는 다음 설정이 포함됩니다.

- 교실 이름
- 일간 칭찬왕 사용 여부
- 주간 칭찬왕 사용 여부
- 월간 칭찬왕 사용 여부
- 학생 메시지 정책
- 학생 로그인 토큰

학생 메시지 정책은 다음과 같이 구분됩니다.

- `disabled` : 메시지 기능 사용 안 함
- `replies_only` : 교사가 시작한 메시지에 학생이 답글만 가능
- `student_initiated` : 학생이 교실 선생님에게 새 메시지 작성 가능

---

### 2. 학생 등록과 관리

교사는 교실에 학생을 등록하고 관리할 수 있습니다.

지원하는 학생 관리 기능은 다음과 같습니다.

- 학생 1명씩 추가
- 여러 학생 일괄 생성
- 학생 이름, 이메일, 성별, 아바타 수정
- 학생 PIN 설정
- 학생 비밀번호 재설정
- 학생 삭제 또는 비활성 처리

학생 일괄 생성은 한 번에 최대 30명까지 처리하도록 제한되어 있습니다. 학생 삭제 시에는 이미 칭찬, 쿠폰, 메시지 등 활동 이력이 있는 경우 데이터를 보존하기 위해 학생 계정을 바로 삭제하지 않고 교실 멤버십을 비활성화하는 방향으로 처리합니다.

---

### 3. 학생 PIN 로그인

학생은 일반 Devise 로그인 화면을 사용하지 않고, 교실별 학생 로그인 화면에서 자신의 이름을 선택한 뒤 4자리 PIN을 입력해 로그인합니다.

학생 로그인 흐름은 다음과 같습니다.

1. 교사가 교실별 학생 로그인 주소나 QR 코드를 안내합니다.
2. 학생은 해당 주소로 접속합니다.
3. 학생은 자기 이름을 선택합니다.
4. 4자리 PIN을 입력합니다.
5. 로그인 후 자기 교실 맥락의 학생 페이지로 이동합니다.

학생 로그인 주소는 교실별 토큰을 사용합니다. 교사가 토큰을 재발급하면 기존 로그인 주소와 QR 코드는 더 이상 사용할 수 없습니다.

학생 세션은 교실 수업 환경을 고려해 일정 시간 동안 활동이 없으면 자동으로 만료되도록 설계되어 있습니다.

---

### 4. 칭찬 기록

교사는 교실 학생에게 칭찬을 줄 수 있습니다. 칭찬을 받은 학생은 포인트가 증가하고, 학생 상세 화면과 교실 화면에서 칭찬 현황을 확인할 수 있습니다.

칭찬 기능은 다음 흐름을 가집니다.

- 교사가 학생에게 칭찬 부여
- 학생의 포인트 증가
- 오늘 받은 칭찬 수 갱신
- 학생 상세 화면 갱신
- 중복 클릭 방지를 위한 짧은 시간 중복 요청 차단

칭찬은 단순한 점수 누적이 아니라, 이후 칭찬왕과 쿠폰 발급 흐름의 기반이 됩니다.

---

### 5. 칭찬왕

교실에서는 일간, 주간, 월간 칭찬왕을 운영할 수 있습니다.

칭찬왕 기능은 다음과 같은 방향으로 설계되어 있습니다.

- 교실 설정에 따라 일간/주간/월간 칭찬왕 사용 가능
- 해당 기간 동안 칭찬을 많이 받은 학생을 집계
- 동점자가 있을 경우 여러 명의 칭찬왕 표시 가능
- 칭찬왕에게 쿠폰을 발급하는 흐름과 연결

교사는 교실 화면에서 칭찬왕 영역을 새로고침하거나, 칭찬왕 기준으로 쿠폰을 발급할 수 있습니다.

---

### 6. 쿠폰 템플릿과 쿠폰 발급

쑥쑥칭찬통장은 칭찬을 실제 보상 경험으로 연결하기 위해 쿠폰 기능을 제공합니다.

쿠폰 템플릿에는 다음 정보가 포함됩니다.

- 쿠폰 이름
- 가중치
- 활성 여부
- 개인 쿠폰 또는 라이브러리 쿠폰 구분
- 기본 이미지 또는 첨부 이미지

관리자는 공통 라이브러리 쿠폰을 만들 수 있고, 교사는 이를 자신의 개인 쿠폰 세트로 가져와 사용할 수 있습니다. 새 교사 계정이 생성될 때 기본 쿠폰을 자동으로 준비하는 구조도 포함되어 있습니다.

쿠폰 발급은 다음 기준으로 구분됩니다.

- 일간
- 주간
- 월간
- 수동
- 하이브리드

쿠폰 발급 시에는 같은 학생에게 같은 기간 기준으로 중복 발급되는 상황을 막기 위한 제약과 중복 요청 방지 로직이 포함되어 있습니다.

---

### 7. 쿠폰 사용 요청과 승인

학생은 자신이 가진 쿠폰을 직접 “사용 완료” 처리하지 않고, 쿠폰 사용 요청을 보냅니다. 교사는 학생 상세 화면에서 요청을 확인하고 승인합니다.

이 흐름은 실제 교실 상황을 반영합니다.

- 학생이 쿠폰 사용 요청
- 교실 화면의 학생 카드에 쿠폰 요청 알림 표시
- 교사가 학생 상세 화면에서 요청 확인
- 교사가 승인하면 쿠폰이 사용 처리됨
- 학생 쿠폰 목록과 교사 관리 화면이 함께 갱신됨

학생이 같은 쿠폰에 대해 여러 개의 대기 중 요청을 만들지 못하도록, 대기 중 요청은 쿠폰당 하나만 허용됩니다.

---

### 8. 학생 메시지

교사와 학생은 교실 맥락 안에서 메시지를 주고받을 수 있습니다.

메시지 기능은 다음 원칙을 따릅니다.

- 현재 단계에서는 교사/관리자와 학생 사이의 메시지만 허용합니다.
- 학생끼리의 메시지는 허용하지 않습니다.
- 답글의 답글은 허용하지 않고, 원글에 대한 한 단계 답글만 허용합니다.
- 학생이 새 메시지를 시작할 수 있는지는 교실 메시지 정책에 따릅니다.
- 학생이 보낸 읽지 않은 메시지는 교실 학생 카드에 알림으로 표시됩니다.

교사가 학생 상세 화면을 열어 메시지를 확인하면, 해당 학생의 읽지 않은 메시지를 읽음 처리하고 교실 카드 알림을 갱신합니다.

---

### 9. 실시간에 가까운 화면 갱신

쑥쑥칭찬통장은 Hotwire 기반으로 화면 일부를 갱신합니다.

주요 갱신 대상은 다음과 같습니다.

- 칭찬 부여 후 학생 상세 정보
- 쿠폰 발급 후 쿠폰 목록
- 학생 쿠폰 사용 요청 후 학생 카드 알림
- 쿠폰 사용 승인 후 학생 쿠폰 목록
- 학생 메시지 확인 후 메시지 알림

교실 화면에는 SNS식 전체 알림 카운트 대신, 학생 카드마다 필요한 알림을 작은 아이콘으로 보여주는 방향을 사용합니다. 이를 통해 교사는 “어떤 학생에게 처리할 일이 있는지”를 교실 화면에서 바로 확인할 수 있습니다.

## 예전 칭찬통장과 웹앱의 차이

| 구분 | 종이 칭찬통장 / 스티커판 | 쑥쑥칭찬통장 |
| --- | --- | --- |
| 칭찬 기록 | 교사가 수동 기록 | 교실 웹 화면에서 즉시 기록 |
| 학생 확인 | 교사가 보여주거나 종이를 확인 | 학생이 직접 로그인해 확인 |
| 보상 관리 | 쿠폰 종이 관리 필요 | 쿠폰 발급/사용 요청/승인 관리 |
| 칭찬왕 | 수동 집계 필요 | 기간별 집계와 화면 표시 |
| 알림 | 교사가 기억해야 함 | 학생 카드별 요청/메시지 알림 |
| 접근 방식 | 교실 안에서만 확인 가능 | 교실별 로그인 주소/QR로 접근 |
| 권한 관리 | 별도 관리 어려움 | 관리자/교사/학생 역할 분리 |

## 설계 원칙

### 교실 맥락을 우선한다

학생, 칭찬, 쿠폰, 메시지는 모두 교실 맥락 안에서 해석됩니다. 같은 학생이라도 어떤 교실에서 접속했는지에 따라 보여야 하는 정보가 달라질 수 있습니다.

### 학생 로그인은 단순해야 한다

학생은 이메일과 비밀번호를 직접 입력하는 대신, 교실별 로그인 주소에서 자기 이름을 선택하고 4자리 PIN으로 접속합니다. 교실 수업 중 빠르게 사용할 수 있도록 하기 위한 선택입니다.

### 교사의 처리 흐름을 줄인다

쿠폰 사용 요청이나 학생 메시지는 전체 알림 카운트보다 학생 카드별 아이콘으로 표시합니다. 교사는 교실 화면에서 어떤 학생을 확인해야 하는지 바로 알 수 있습니다.

### 기록은 가능한 한 보존한다

학생에게 이미 칭찬, 쿠폰, 메시지 같은 활동 이력이 있으면 삭제 대신 비활성화를 우선 고려합니다. 교실 운영 기록을 보존하기 위해서입니다.

### Hotwire로 필요한 부분만 갱신한다

전체 페이지를 매번 새로고침하지 않고, 칭찬, 쿠폰, 메시지 알림처럼 변경이 필요한 영역을 중심으로 갱신합니다.
