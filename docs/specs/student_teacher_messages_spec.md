# specs/student_teacher_messages_spec.md

# 쑥쑥칭찬통장 — 교사/admin ↔ 학생 메시지 기능 Spec (Phase 1)

## 목적

학생용 자기 페이지와 교사용 학생 관리 페이지가 분리된 상태에서,
교사 또는 관리자가 특정 학생에게 칭찬/안내 메시지를 보내고,
학생은 자기 페이지에서 해당 메시지를 확인한 뒤
자기에게 메시지를 보낸 상대에게 응답할 수 있도록 한다.

이번 단계의 핵심은 **범용 notification 시스템**을 만드는 것이 아니라,
현재 허용된 사용자 조합 안에서의 **직접적인 메시지 흐름**을 먼저 작고 명확하게 구현하는 것이다.

---

## 이번 단계에서 구현할 기능

### 포함

1. 교사/admin이 학생에게 메시지 전송
2. 학생이 자기 페이지에서 받은 메시지 확인
3. 학생이 자기에게 메시지를 보낸 상대에게 응답 메시지 전송
4. 교사/admin이 학생 상세 페이지에서 해당 학생과의 메시지 확인
5. teacher/admin 원글과 학생 답글이 댓글처럼 보이도록 최소 구조 제공
6. 전체 페이지 reload 대신 부분 갱신으로 새 답글이 바로 보이도록 한다

### 제외

이번 단계에서는 아래는 구현하지 않는다.

1. 범용 Notification 시스템
2. 읽음/안읽음 고급 상태 관리
3. 첨부파일
4. 다자간 대화 / 학생↔학생 메시지
5. 여러 참여자를 묶는 복잡한 thread 모델
6. 실시간 웹소켓 채팅
7. 검색 / 보관 / 삭제 / 신고 기능

---

## 핵심 사용자 흐름

### 1. 교사/admin → 학생

- 교사 또는 관리자가 학생 상세 페이지에 들어간다
- “메시지 보내기” 입력 영역에 내용을 작성한다
- 전송하면 해당 학생에게 메시지가 생성된다
- 학생은 자기 페이지에서 메시지를 볼 수 있다

### 2. 학생 → 발신자 응답

- 학생이 자기 페이지에서 받은 메시지를 확인한다
- 학생은 자기에게 먼저 메시지를 보낸 상대에게만 응답할 수 있다
- teacher/admin 원글 바로 아래의 응답 입력창에 답장을 작성한다
- 전송하면 해당 teacher/admin 원글 카드 안에 학생 답글이 바로 추가된다
- 전송한 답글은 해당 teacher/admin이 학생 상세 페이지에서 같은 원글 카드 안에서 확인할 수 있다

중요:
- 이번 단계에서는 학생이 **아무 교사/admin에게나 새 메시지를 시작하는 구조가 아니라**
  **자기에게 온 메시지에 대해 응답하는 흐름**을 우선으로 한다.
- 즉, 초기 모델은 “자유 채팅”보다는 **교사/admin 시작형 원글 + 학생 댓글형 응답 흐름**에 가깝다.

---

## 설계 원칙

1. Rails way 우선
2. 범용 notification 으로 과하게 일반화하지 않기
3. 도메인 모델을 명확하게 두기
4. 학생 페이지와 교사용 페이지의 역할을 섞지 않기
5. 나중에 쿠폰 요청 기능과 결합 가능하도록 하되, 이번 단계에서는 메시지 자체에 집중

---

## 도메인 모델 방향

이번 단계에서는 별도의 메시지 모델을 직접 만든다.

예상 후보:
- `UserMessage`
- 또는 현재 저장소 네이밍과 충돌하지 않는 동등한 중립적 이름

권장 방향:
- 이번 단계의 UI는 교사/admin ↔ 학생 메시지에 한정하되,
  모델 이름은 향후 사용자 간 메시지 확장을 버틸 수 있도록 과하게 좁히지 않는다.
- 다만 범용 notification system 으로 일반화하지는 않는다.

### 최소 필드 예시

- sender_id
- recipient_id
- classroom_id
- body
- parent_message_id
- created_at

추가 검토 가능:
- read_at

원칙:
- teacher/admin 원글은 `parent_message_id = nil`
- 학생 답글은 `parent_message_id = teacher/admin 원글 id`
- 이번 단계에서는 답글의 답글 같은 다단계 thread 는 만들지 않는다

---

## 메시지 관계의 기본 규칙

### 보낼 수 있는 사람

- teacher/admin → 특정 학생에게 메시지 전송 가능
- student → 자기에게 먼저 메시지를 보낸 teacher/admin에게만 응답 가능

### 보낼 수 없는 것

- 학생 → 다른 학생에게 메시지 전송 불가
- 학생 → 임의의 다른 교사/admin에게 새 메시지 시작 불가
- 학생 → 다른 학생의 메시지 열람 불가
- teacher/admin ↔ teacher/admin 메시지는 이번 단계에서 구현하지 않음
- 교사 → 담당 범위를 벗어난 학생에게 전송 가능 여부는 현재 정책 구조를 먼저 읽고 결정

주의:
- 교사의 담당 학생 범위 제한은 현재 classroom/roles/policy 구조를 확인한 뒤,
  **기존 권한 원칙에 맞게 최소 변경으로 반영**한다.
- 관리자는 teacher와 유사한 전송/열람 권한을 둘 수 있으나,
  이번 단계에서는 학생 page와 managed student page 안에서만 사용한다.

## 페이지별 역할

### A. 교사용/관리자용 학생 상세 페이지

목적:
- 특정 학생과의 메시지 확인 및 새 메시지 전송

이번 단계에서 포함할 UI 후보:
- 학생과의 최근 메시지 목록
- 메시지 입력 폼
- 전송 버튼

### B. 학생 자기 페이지

목적:
- teacher/admin에게서 온 메시지 확인
- 응답 작성

이번 단계에서 포함할 UI 후보:
- teacher/admin 원글 카드 목록
- 각 원글 카드 아래의 학생 답글 목록
- 각 원글 카드 아래의 응답 입력 폼
- 전송 버튼

중요:
- 학생 페이지는 “내 메시지”만 보여야 한다
- 다른 학생 메시지, 전체 교실 메시지 같은 개념은 넣지 않는다
- 학생은 자기에게 먼저 메시지를 보낸 상대에게만 응답할 수 있어야 한다
- 학생이 답글을 여러 번 달아도 같은 원글 카드 안에 누적되어야 한다

## 구현 범위에 대한 구체 결정

### 1. thread 모델을 크게 만들지 말 것

이번 단계에서는 Slack/채팅앱 같은 복잡한 대화 시스템을 만들지 않는다.

가능하면 아주 얕은 댓글형 구조로 시작한다.

- teacher/admin 이 학생에게 보낸 메시지는 원글
- 학생 응답은 그 원글 아래에 붙는 답글
- teacher/admin 여러 명이 같은 학생에게 각각 메시지를 보내면,
  원글 카드가 여러 개 생기고 학생 답글은 각 원글 카드 안에 붙는다

핵심은:
**teacher/admin이 원글을 남기고, 학생이 그 원글 아래에 답글을 달 수 있으면 충분하다.**

---

### 2. 학생은 “새 대화 시작”보다 “응답” 중심

이번 단계의 UX는 아래에 가깝다.

- teacher/admin이 먼저 보냄
- 학생은 해당 발신자에게 응답

나중에 필요하면 학생 시작형 메시지나 학생 간 메시지를 추가할 수 있지만,
이번 단계에서는 범위를 좁혀서 구현한다.

---

### 3. 알림은 아직 메시지 목록/뱃지 수준으로만 고려

이번 단계에서는 범용 notification 모델을 만들지 않는다.

대신 필요하다면 이후 단계에서 아래 정도는 검토 가능하다.

- navbar에 미확인 메시지 수
- 학생 페이지/교사 페이지에서 새 메시지 표시

하지만 이번 spec 구현의 필수 요구사항은 아니다.

---

## 컨트롤러 / 라우트 방향 가이드

구체 구조는 저장소를 먼저 읽고 결정하되, 아래 원칙을 따른다.

1. 불필요하게 새 리소스를 많이 만들지 말 것
2. 그러나 메시지 도메인이 명확하다면 resource 분리는 허용
3. teacher-managed student page 와 student self page 양쪽에서 재사용 가능한 형태를 우선 고려
4. authorization 은 controller/policy 기준으로 명확히 둘 것
5. 뷰에서 권한 분기 남발하지 말고 controller/helper/local 로 정리할 것
6. 새 범용 inbox 페이지를 만들기보다 기존 self page / managed student page 안에 섹션으로 넣는 방향을 우선 고려할 것
7. 새 메시지/답글 생성 후에는 가능하면 Turbo Stream 등으로 메시지 섹션만 부분 갱신하는 방향을 우선 고려할 것

가능한 방향 예시:
- nested resource 형태
- 학생 페이지용과 교사용 전송 경로를 분리
- 또는 messages controller 하나에 권한 기반 분기

단, 실제 저장소 구조를 읽고 가장 덜 무리한 방식을 택한다.

## 정책 / 권한 요구사항

이번 단계에서 반드시 지켜야 할 것:

1. 학생은 자기 메시지만 볼 수 있어야 한다
2. 학생은 자기에게 먼저 메시지를 보낸 상대에게만 응답할 수 있어야 한다
3. 교사는 허용된 학생에게만 메시지를 보낼 수 있어야 한다
4. 관리자도 메시지 열람/전송이 가능한지 정책으로 명확히 할 것
5. URL 직접 접근으로 다른 학생 메시지 열람이 가능하면 안 된다
6. 학생은 teacher/admin 간 메시지나 다른 학생 간 메시지에 접근할 수 없어야 한다

## 데이터 표시 방식

이번 단계에서는 최소한 아래만 만족하면 된다.

- teacher/admin 원글과 학생 답글의 구조가 구분 가능
- 시간 순 정렬
- 발신자 구분 가능
- 메시지 본문 확인 가능
- 학생 답글은 자신이 답한 teacher/admin 원글 카드 안에 표시

추가로 가능하면 좋지만 필수는 아님:
- teacher/admin 원글과 학생 답글의 들여쓰기/보더/배경 차이
- 최근 메시지가 아래에 오도록 정렬
- 간단한 빈 상태 메시지

주의:
- 이번 단계에서는 강한 채팅앱 스타일보다 댓글형 카드 구조를 우선한다.
- 색상만으로 역할을 구분하기보다 카드 구조, 아바타, 이름, 역할 라벨로 구분하는 편을 우선 검토한다.

---

## 실패/예외 처리

이번 단계에서 최소한 아래는 처리한다.

1. 빈 메시지는 전송되지 않아야 한다
2. 권한 없는 사용자의 전송 시도는 차단되어야 한다
3. 권한 없는 메시지 열람 시도는 차단되어야 한다
4. 존재하지 않는 대상/잘못된 대상에 대한 전송은 안전하게 실패해야 한다
5. 학생이 다른 사람의 원글에 답글을 달려고 하면 차단되어야 한다
6. 학생이 답글의 답글을 다는 구조는 이번 단계에서 허용하지 않는다

## Codex 작업 지시

다음 원칙으로 작업한다.

1. 먼저 저장소를 읽고,
   - 현재 분리된 학생 self page / 관리자용 학생 page 구조
   - users 관련 routes/controllers/views
   - existing policy 구조
   - teacher/admin/student 역할 분기
   를 확인한다.

2. 바로 구현하지 말고,
   **변경 계획(diff 요약)** 부터 제시한다.

3. 구현은 작은 단계로 나눈다.

권장 작업 순서:

- Step A. 메시지 도메인 모델 설계 제안
- Step B. 권한/정책 정리
- Step C. 교사용 학생 페이지 메시지 UI
- Step D. 학생 자기 페이지 댓글형 메시지 UI
- Step E. 테스트 추가

4. 과한 범용화 금지
5. notification system 으로 확장하려 하지 말 것
6. 이번 단계에서는 메시지 기능 자체를 작게 완성하는 데 집중할 것

## Codex가 먼저 확인해야 할 질문

1. 현재 teacher/student 분리된 users/show 구조가 어떤 방식으로 구현되어 있는가?
2. 메시지 기능을 붙이기에 가장 자연스러운 화면 위치는 어디인가?
3. 기존 role/policy 구조에서 teacher 가 어떤 학생까지 다룰 수 있는가?
4. 학생 응답을 “자유 새 메시지”가 아니라 “teacher/admin 원글에 대한 댓글”로 제한하려면 어느 계층에서 막는 것이 가장 자연스러운가?
5. 메시지 모델을 댓글형 구조로 시작하려면 어떤 속성만 있으면 충분한가?

- 학생이 teacher/admin 원글 아래에 답글을 달 수 있다
- 메시지 접근은 역할/권한에 맞게 제한된다
- 나중에 쿠폰 사용 요청 기능과 병렬로 유지할 수 있는 작은 메시지 도메인이 생긴다
