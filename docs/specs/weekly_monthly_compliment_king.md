# 작업 명세: 주간/월간 칭찬왕 지원

## 배경

현재 `suksuk_praise`의 상태는 다음과 같습니다.

- 일간 칭찬왕은 이미 구현되어 있다.
- 현재 일간 로직은 다음 위치에 분산되어 있다.
  - `ClassroomsController#show`
  - `ClassroomsController#refresh_compliment_king`
  - `CouponDraw::Issue`
- 교실 상세 화면 UI는 현재 "오늘의 칭찬왕"만 표시한다.
- `UserCoupon`에는 이미 기간 관련 구조(`issuance_basis`, `period_start_for`)가 일부 존재하고 `weekly`도 포함되어 있지만, 실제 앱 동작은 아직 사실상 일간 전용이다.
- `monthly` 지원은 아직 완전히 구현되어 있지 않다.
- `Classroom`에는 주간/월간 칭찬왕 기능을 켜고 끄는 설정 필드가 아직 없다.

## 목표

다음을 구현한다.

1. 일간 / 주간 / 월간 칭찬왕 계산을 공통 구조로 정리
2. 교실 단위에서 주간/월간 칭찬왕 기능을 켜고 끌 수 있는 설정 추가
3. 교사용 관리 UI에서 주간/월간 사용 여부를 토글할 수 있도록 구현
4. 기존 일간 동작은 유지

## 중요한 제약

- 일반적인 Rails 관례를 우선 따른다.
- 불필요한 추상화는 도입하지 않는다.
- 기존 일간 동작을 깨뜨리지 않는다.
- period 관련 로직을 controller에 계속 남기기보다, 중복된 일간 로직을 작은 service/query object로 추출하는 방향을 선호한다.
- 단순히 라벨이나 UI 문구만 바꾸는 방식으로 끝내지 않는다.
- 기능 토글은 화면에서만 보이는 장식이 아니라 도메인 레벨에서도 실제로 동작해야 한다.
- 현재 프로젝트의 authorization 스타일과 일관성을 유지한다.
- 변경은 점진적이고 이해 가능한 범위로 유지한다.

## 요청하는 구현 방향

### 1. 칭찬왕 집계 로직 추출

교실과 기간 단위를 받아 칭찬왕을 계산하는 작은 service 또는 query object를 만든다.

기대 동작:

- `daily`: 오늘
- `weekly`: 이번 주 (월요일 시작 기준)
- `monthly`: 이번 달

결과는 controller에서 단순하게 사용할 수 있어야 한다. 예를 들면 다음 정도면 충분하다.

- winners
- compliment_count
- 필요하다면 period label 또는 period key

현재 중복된 일간 로직이 들어 있는 아래 위치들은 공통 구조를 사용하도록 리팩토링한다.

- `ClassroomsController#show`
- `ClassroomsController#refresh_compliment_king`

### 2. 교실 설정 토글 추가

칭찬왕 기간 단위를 켜고 끌 수 있는 boolean 설정을 `Classroom`에 추가한다.

최소한 다음 필드는 포함한다.

- `daily_compliment_king_enabled`
- `weekly_compliment_king_enabled`
- `monthly_compliment_king_enabled`

권장 기본값:

- daily: true
- weekly: false
- monthly: false

만약 기존 프로젝트 스타일에 맞춰 이름을 더 단순하게 바꾸는 편이 낫다면, 의미가 충분히 명확한 Rails 스타일 이름으로 조정해도 된다.

### 3. 교사/교실 관리 UI

교사가 교실 관리 또는 수정 화면에서 주간/월간 사용 여부를 조정할 수 있도록 UI를 추가한다.

요구사항:

- 주간 토글
- 월간 토글
- 일간은 기본적으로 켜진 상태를 유지한다. 특별한 이유가 없다면 일단 그대로 둔다.

UI는 단순하게 유지하고, 기존 Tailwind 기반 스타일과 일관되게 맞춘다.

### 4. 쿠폰 발급 규칙 정리

`CouponDraw::Issue`를 검토하고, 새 기간 구조와 맞게 정리한다.

현재 문제점:

- `daily_top` 로직이 일간 기준으로 하드코딩되어 있다.
- 기간 중복 발급 가드도 사실상 일간 전용이다.
- `normalize_basis`가 대부분의 값을 daily/manual로 축소하고 있다.

이번 작업에서는 다음을 만족한다.

- 기존 일간 쿠폰 발급 흐름은 유지한다.
- 주간/월간 basis 확장에 대비할 수 있도록 코드 구조를 정리한다.
- 적절한 범위에서 `monthly`를 추가한다.
- 필요하지 않은 복잡한 새 쿠폰 정책은 만들지 않는다.
- 주간/월간 쿠폰 발급을 아직 비활성 상태로 둘 계획이라면, 그 점이 코드에서 명확하게 드러나고 이후 확장도 쉽게 할 수 있도록 정리한다.

즉,

- 주간/월간 칭찬왕 "표시" 지원은 이번 작업 범위에 포함한다.
- 주간/월간 쿠폰 발급은 단계적으로 보류할 수 있지만, 현재 코드가 오해를 부르거나 반쯤 하드코딩된 상태로 남아 있지는 않게 정리한다.

### 5. 테스트

다음 항목에 대한 테스트를 추가하거나 갱신한다.

- 일간 집계가 기존처럼 정상 동작한다.
- 주간 집계는 월요일 시작 기준을 사용한다.
- 월간 집계는 이번 달 기준으로 동작한다.
- 주간/월간 설정이 비활성일 때 관련 기능이 숨겨지거나 차단된다.
- 기존 일간 동작이 유지된다.

이 영역에서 이미 강한 system test 패턴이 있는 것이 아니라면, 과도하게 넓은 system test보다는 model/service/request/controller spec 같은 집중된 테스트를 우선한다.

- 여기저기 흩어진 magic string
- 서버 측 가드 없이 화면에만 있는 가짜 토글
- 이번 범위를 벗어나는 쿠폰/템플릿 로직의 대규모 재작성
- 범위를 넘는 광범위한 리팩토링
